### Architecture Decision: Modular Monolith, Not Microservices

For a startup team (1-3 devs), a **modular monolith** is recommended. The code is organized as independent modules (services) within one FastAPI process, sharing the same DB, but each module has clean boundaries. This means:

- Zero inter-service networking overhead
- Single deployment unit early on
- Can extract into true microservices later by splitting along existing module boundaries
- All modules share connection pools (PostgreSQL, Redis)
