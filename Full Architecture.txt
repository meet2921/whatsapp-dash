## 1. System Overview

A **modular monolith** — single deployable FastAPI process with strictly bounded modules. Extracted to microservices later if needed, without restructuring.
  
**Core principles:**
- Every table has `org_id` (multi-tenant isolation, not optional)
- All operations idempotent (safe to retry anywhere in the stack)
- Money operations are two-phase (reserve → confirm/rollback), never fire-and-forget
- Webhooks: receive fast (< 200ms), process slow (async workers)
- Every log line carries `request_id` + `org_id` from day one

## 3. Multi-Tenant Isolation Design

```
Every DB table:
  org_id  UUID  NOT NULL  INDEX

Every API request:
  JWT → decoded → org_id injected into request context
  SQLAlchemy event listener: auto-appends WHERE org_id = :current_org on all queries
  Middleware: sets org_id in ContextVar for logging

Tenant boundaries:
  Organizations → Users → WABAAccounts → PhoneNumbers
  Contacts, Conversations, Messages, Campaigns, Flows
  Wallets, Transactions, Invoices — all scoped to org_id
```

**Row-Level Security (optional, Phase 2):** Enable PostgreSQL RLS policies per table as second layer of defense.

## 5. Upgrade 1 — Global Idempotency System

### Problem
Without idempotency: retries cause double-sends, double-billing, duplicate webhook processing.

### Design

```
Layer 1 — Redis (fast path):
  Key pattern: idem:{namespace}:{sha256(parts)}
  TTL: 24 hours
  Operation: SET NX EX (atomic check-and-set)
  On hit: return "already processed" immediately

Layer 2 — DB unique constraint (durable fallback):
  wallet_transactions.idempotency_key UNIQUE(org_id, key)
  messages.idempotency_key UNIQUE
  webhook_events.idempotency_key UNIQUE(source, key)
  On constraint violation: treat as duplicate, return success
```

### Idempotency Key Derivation

| Operation | Key Components |
|---|---|
| Webhook event | `source + Meta entry_id + event_type` |
| Wallet credit | `org_id + razorpay_payment_id` |
| Wallet debit | `org_id + message_id` |
| Message send | `org_id + phone + campaign_id + content_hash` |
| Campaign dispatch | `campaign_id + recipient_phone` |

### Flow
```
Request arrives with idempotency key
  → Check Redis: key seen? → return cached result (HTTP 200)
  → Not seen: SET NX in Redis, proceed
  → On success: write result to Redis value
  → On DB unique violation: idempotency satisfied, return OK
  → On failure: DELETE Redis key (allow retry)
```

## 6. Upgrade 2 — Atomic Billing (Two-Phase Credit System)

### Problem
Fire-and-forget billing: message fails after deduction → client loses money. Or parallel requests → balance goes negative.

### States
```
pending     → reserved, message not yet confirmed
confirmed   → message sent, credit consumed
rolled_back → message failed, credit refunded
```

### Flow
```
1. RESERVE phase (DB transaction, row lock on wallets):
   a. Lock wallet row (SELECT FOR UPDATE)
   b. Check balance ≥ amount → else 402 error
   c. INSERT wallet_transaction (status=pending)
   d. UPDATE wallets.balance -= amount
   e. Commit

2. SEND phase:
   → Call Meta API to send message

3a. SUCCESS → CONFIRM:
   → UPDATE wallet_transaction SET status=confirmed

3b. FAILURE → ROLLBACK:
   → UPDATE wallets.balance += amount (inside new transaction)
   → UPDATE wallet_transaction SET status=rolled_back
   → Delete Redis idempotency key (allow retry)
```

### Concurrent Safety
- `SELECT FOR UPDATE` on wallets row prevents concurrent over-deduction
- `UNIQUE(org_id, idempotency_key)` on transactions prevents double-reserve
- All balance math happens inside PostgreSQL transactions — no Python arithmetic on fetched values

## 7. Upgrade 3 — Celery Queue Segregation

### Queue Architecture
```
campaign_queue   → Bulk message dispatch (high volume, rate-limited)
message_queue    → Single message sends (low latency priority)
webhook_queue    → Inbound event processing (must not back up)
billing_queue    → Razorpay events, invoice gen, balance alerts
media_queue      → S3 upload + metadata extraction (CPU-bound)
analytics_queue  → Stats aggregation (low priority)
```

### Why Segregate?
- Campaign dispatch (thousands of msgs) must never block webhook processing
- Billing tasks run with `prefetch=1` — never batch-process payments
- Media tasks are CPU/IO-bound — isolated to prevent starving message workers
- Webhook queue: dedicated worker always running, no competition

### Retry Policies Per Queue
```
campaign_queue:
  max_retries: 3
  backoff: exponential (30s → 60s → 120s)
  acks_late: TRUE (message stays in queue until task completes)

message_queue:
  max_retries: 5
  backoff: exponential (10s → 30s → 90s → 270s → 810s)
  acks_late: TRUE

webhook_queue:
  max_retries: 5
  backoff: fixed 60s (Meta expects consistent retry timing)
  acks_late: TRUE

billing_queue:
  max_retries: 3
  backoff: exponential (60s → 180s → 540s)
  prefetch: 1 (never parallel-process billing)
  acks_late: TRUE

media_queue:
  max_retries: 3
  backoff: exponential
  acks_late: TRUE
```

### Dead Letter Handling
- After max_retries: task moved to `{queue}_dlq` (dead letter queue)
- Celery Beat scans DLQs every hour → admin notification
- Admin can inspect and requeue from admin panel

## 8. Upgrade 4 — Webhook Reliability System

### Problem
Meta requires HTTP 200 within 5 seconds. Processing takes longer. Any error = Meta retries 3× = duplicate processing risk.

### Design
```
Step 1 — RECEIVE (< 200ms):
  → Verify X-Hub-Signature-256 (HMAC-SHA256)
  → Extract Meta's entry.id as idempotency key
  → Check Redis idempotency (skip if already seen)
  → INSERT into webhook_events (processed=FALSE)
  → Return HTTP 200 immediately

Step 2 — PROCESS (async, Celery):
  → Worker picks up event by ID
  → Parse payload: message / status_update / template_status
  → Dispatch to correct handler:
      inbound message → create contact/conversation/message
      status update   → update message.status + campaign counts
      template status → update template.status + notify org
  → On success: UPDATE webhook_events SET processed=TRUE
  → On failure: UPDATE retry_count++, last_error=...
               → retry via Celery backoff

Step 3 — BEAT RETRY SCANNER (every 60s):
  → SELECT * FROM webhook_events WHERE processed=FALSE AND retry_count < 5
  → Re-enqueue each (Celery already handles backoff)
  → Events with retry_count >= 5: alert admin
```

### Idempotency in Webhook Processing
- `webhook_events.idempotency_key` unique constraint: DB-level guarantee
- Redis check: fast-path dedup before DB write
- Each handler is idempotent: "upsert contact" not "insert contact"

## 10. Upgrade 6 — Conversation Locking

### Problem
Human agent has conversation open. Bot automation also triggers a reply. Client gets two messages — one from bot, one from agent. Looks broken.

### Lock Architecture
```
Two-layer lock:

Layer 1 — Redis (fast, auto-expire):
  Key:   conv_lock:{conversation_id}
  Value: user_id
  TTL:   300 seconds (5 minutes)
  SET NX EX — atomic acquire

Layer 2 — DB state (durable):
  conversations.locked_by      UUID → users.id
  conversations.locked_at      TIMESTAMPTZ
  conversations.lock_expires_at TIMESTAMPTZ
```

### Lock Flow
```
Agent opens conversation in inbox:
  → Acquire Redis lock (SET NX EX 300)
  → Update DB: locked_by=user_id, lock_expires_at=now+5min
  → WebSocket event: notify other agents "conversation locked by [name]"

Bot automation triggers:
  → automation_service checks: is_locked_by_human(conversation_id)?
    → Check Redis first (fast)
    → Fallback to DB (lock_expires_at > NOW())
  → If locked: SKIP bot reply (bot stays silent)
  → If not locked: bot can reply

Agent closes conversation:
  → Release Redis key (Lua script: check value before delete)
  → Update DB: locked_by=NULL, locked_at=NULL

Auto-expiry:
  → Redis TTL expires automatically (no orphaned locks)
  → Beat task every 5 min: clear DB rows where lock_expires_at < NOW()

Edge cases:
  → Reload/crash: lock auto-expires (TTL)
  → Agent reconnects within TTL: RE-acquire (extend TTL)
  → Two agents same conversation: second SET NX fails → shown "locked by [name]"
```

## 11. Upgrade 7 — Async Media Processing

### Problem
Large file uploads block the API thread. S3 upload can take seconds. Metadata extraction (image dimensions, video thumbnails) is CPU-bound.

### Flow
```
Step 1 — UPLOAD (< 100ms response):
  POST /api/v1/media/upload  (multipart)
  → Save file to /tmp/{uuid}_{filename}
  → INSERT media_assets (status=pending)
  → Enqueue process_media_upload.delay(asset_id, temp_path)
  → Return: {"asset_id": "...", "status": "pending"}

Step 2 — PROCESS (Celery, media_queue):
  → UPDATE status=processing
  → Upload to S3: s3://bucket/media/{org_id}/{asset_id}/{filename}
  → Extract metadata:
      image: width, height, format (Pillow)
      video: duration, codec (ffprobe)
      document: page_count (pdfplumber)
  → UPDATE status=completed, s3_key=..., public_url=..., meta={...}
  → DELETE temp file
  → On error: UPDATE status=failed, error_message=...
             → Celery retry (max 3)

Step 3 — POLL (Frontend):
  GET /api/v1/media/{asset_id}/status
  → Returns: {status, url, error}
  → Frontend polls every 1s until status=completed or failed
  → Or: WebSocket event pushed on completion (preferred)
```

### S3 Key Structure
```
media/{org_id}/{asset_id}/{original_filename}

Access pattern:
  Public CDN URL: https://cdn.tiercemsg.com/media/{org_id}/{asset_id}/{filename}
  Or presigned S3 URL (for private files)
```

## 12. Bonus Features

### Redis Distributed Lock (Cross-Service)
```
Use cases:
  Campaign dispatch: prevent double-launch of same campaign
  Template submission: prevent duplicate Meta API calls
  Wallet recharge: prevent concurrent Razorpay order creation

Pattern:
  Lock key:  lock:{resource}:{id}
  Lock value: {worker_uuid}         (must own the lock to release)
  TTL:       30 seconds             (auto-releases on crash)
  Acquire:   SET NX EX (atomic)
  Release:   Lua script (check value, then delete — atomic)
  Retry:     3 attempts, 500ms apart → raise if cannot acquire
```

### Per-Org API Rate Limiting
```
Algorithm: Sliding window (Redis sorted set)

Key:    ratelimit:{action}:{org_id}
Entry:  score=timestamp, member=timestamp_nanoseconds

Per action limits (configurable per plan):
  api_calls:        1000 / minute
  message_send:     500 / minute
  campaign_create:  10 / hour
  csv_upload:       5 / hour

Meta send rate:
  meta_send:{phone_number_id}:  80 / second  (Meta hard limit)

On exceed: HTTP 429 + Retry-After header
```

### Tenacity Retry Decorator
```
Applied to:
  Meta API calls:     5 retries, exponential 2s→30s, on HTTP 5xx + ConnectError
  Razorpay API calls: 3 retries, exponential 1s→10s
  S3 uploads:         3 retries, exponential 2s→20s
  DB operations:      3 retries, exponential 0.5s→5s (transient PG errors)

NOT applied to:
  Billing deductions (idempotency handles retry safety)
  Webhook handler (Celery handles retry)
```

## 13. WhatsApp API Integration

### Meta Cloud API vs BSP Decision Matrix

| Factor | Meta Cloud API (Direct) | BSP (Twilio, 360Dialog, Gupshup) |
|---|---|---|
| Cost per conversation | Base Meta price | Meta price + 20–50% BSP markup |
| Setup time | 1–2 weeks (business verification) | Same day |
| Webhook control | Full, real-time | BSP-filtered |
| Template management | Direct Meta API | Through BSP portal |
| Support | Meta (slow) | BSP (faster) |
| Rate limits | Meta limits | Same (BSP doesn't add capacity) |
| Scalability | Unlimited (with Meta approval tiers) | Capped by BSP plan |

**Recommendation: Meta Cloud API directly.**

Reason: You're reselling — every BSP markup percentage is direct margin loss. At scale, even 20% BSP markup on ₹2/msg = ₹0.40/msg lost on every marketing message sent by every client.

### WhatsApp Pricing (India, Mar 2026)
| Conversation Type | Meta Cost (USD) | Suggested Selling Price (INR) | Margin |
|---|---|---|---|
| Marketing | ~$0.0117 | ₹2.00 | ~105% |
| Utility | ~$0.0040 | ₹0.80 | ~140% |
| Authentication | ~$0.0034 | ₹0.60 | ~124% |
| Service (inbound 24hr) | $0.00 | ₹0.00 | — |

Note: 1 conversation = any number of messages within 24-hour window. Not per-message.

### Message Send Flow
```
POST /api/v1/messages/send
  → Auth + tenant context
  → Rate limit check (per org, per phone number)
  → Idempotency check
  → Billing: reserve_credits (two-phase begin)
  → whatsapp_service.send_template()
      → httpx.post graph.facebook.com/v22.0/{phone_id}/messages
      → Tenacity retry on 5xx/network error
  → Save message to DB (wa_message_id, status=sent)
  → Billing: confirm_credits
  → Publish to Redis pub/sub → WebSocket → live inbox update
  → Return {message_id, wa_message_id}
```

### Inbound Message Flow
```
Meta POST → /api/v1/webhooks/meta
  → Verify HMAC-SHA256 signature
  → Idempotency check (Redis + DB)
  → Save webhook_event (processed=FALSE)
  → Return HTTP 200

Celery (webhook_queue):
  → Parse payload type: message | status | template_status
  → If message:
      Upsert contact (phone → org's contacts)
      Find/create conversation (check 24hr session)
      Save message to messages table
      Check automation triggers (keyword match → flow)
      If conversation locked_by human: skip bot
      Else: check flows for auto-reply
      Publish to Redis → agent inbox WebSocket
  → If status update:
      Update messages.status, delivered_at, read_at
      Update campaign_recipients.status
      Update campaigns.delivered_count / read_count
  → Mark webhook_event processed=TRUE
```

## 14. Billing System Detail

### Margin Management (Admin Panel)
```
Admin sets in credit_rates table:
  conversation_type = MARKETING
  country_code = IN
  meta_cost_usd = 0.0117     ← updated monthly from Meta invoice
  selling_price_inr = 2.00   ← what clients pay
  usd_to_inr_rate = 83.5     ← updated weekly

System auto-calculates margin_percent for display.
Admin can set per-tenant overrides (enterprise discounts):
  org_id = specific_client → custom selling_price_inr
```

### Razorpay Integration Flow
```
Client clicks "Add ₹1000":
  → POST /billing/recharge {amount: 1000}
  → Create Razorpay order (amount in paise)
  → Return order_id to frontend

Frontend:
  → Razorpay checkout widget
  → Client pays
  → Razorpay fires payment.captured webhook

Backend (billing_queue):
  → Verify Razorpay webhook signature (HMAC-SHA256)
  → Idempotency check: org_id + razorpay_payment_id
  → Credit wallet (two-phase: idempotent credit)
  → Generate invoice (PDF via WeasyPrint)
  → Email invoice to org_admin
  → Push WebSocket event: "Wallet credited ₹1000"
```

## 16. Security Design

| Area | Mechanism |
|---|---|
| Multi-tenant isolation | `org_id` on every query, SQLAlchemy event listener auto-appends WHERE |
| API auth | JWT (HS256), 15min access token + 7-day refresh |
| RBAC | `user_role` enum; `Depends(require_role(...))` on protected routes |
| Webhook signatures | HMAC-SHA256 verified before any processing (Meta + Razorpay) |
| Token storage | WABA access tokens: Fernet-encrypted at rest in DB |
| Secrets | AWS Secrets Manager (never in .env in prod) |
| Rate limiting | Nginx (global) + Redis sliding window (per-org) |
| SQL injection | SQLAlchemy ORM + parameterized queries only — no raw SQL |
| Password | bcrypt, cost factor 12 |
| Audit | All admin actions → audit_log table |
| PDPA/GDPR | Opt-in tracking, contact deletion cascade, data export endpoint |
| PII | Phone numbers searchable but stored with org-level encryption |
